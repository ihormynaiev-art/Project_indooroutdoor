## Описание проекта

- ✅ Форма авторизации (Login)

- ✅ Форма регистрации Homeowner (Homeowner SignUp)
- ✅ Профиль подрядчика (Contractor Profile Setup)
- ✅ Навигация и Explanation страница
- ✅ Загрузка лицензий и страховок (Upload License & Insurance)
- ✅ Настройки отзывов (Reviews Settings)
- ✅ Запросы на котировки (Request Quotes)
- ✅ Сообщения провайдера (Messages)
- ✅ Настройки аккаунта (Account Settings)
- ✅ Переключатель меню (Menu Toggle)
- ✅ Главная страница (Home Page)
- ✅ Валидация полей форм
- ✅ Безопасность (XSS, SQL-инъекции)
- ✅ UI/UX проверки (адаптивность, доступность, производительность)

## Установка

### Предварительные требования:
- Node.js (версия 18 или выше)
- npm или yarn

### Шаги установки:

1. Клонируйте репозиторий (если используется Git):
```bash
git clone <repository-url>
cd indooroutdoor
```

2. Установите зависимости:
```bash
npm install
```

3. Установите браузеры Playwright:
```bash
npx playwright install
```

4. (Опционально) Установите системные зависимости:
```bash
npx playwright install --with-deps
```

## 🏗 Архитектура тестов

Проект использует **Functional Testing Architecture** с элементами:
- **BDD-стиль**: использование `test.step()` для структурирования
- **Упрощенный Page Object Pattern**: вспомогательные функции на уровне модуля
- **Гибкие селекторы**: адаптивный поиск элементов

### Структура теста:
```typescript
test.describe('Suite Name', () => {
  test.beforeEach(async ({ page }) => {
    // Подготовка перед каждым тестом
  });

  // ОБЯЗАТЕЛЬНО: Закрытие браузера после каждого теста
  test.afterEach(async ({ page, context }) => {
    await page.close();
    await context.close();
  });

  test('Test Name', async ({ page }) => {
    await test.step('Step description', async () => {
      // Действия теста
    });
  });
});
```

**Примечание**: В Playwright браузеры обычно закрываются автоматически, но явное закрытие через `test.afterEach` гарантирует освобождение ресурсов и является лучшей практикой.

## 🎨 UI/UX и Адаптивность - Обязательные проверки

**Важно**: Все тесты должны включать проверки UI/UX и адаптивности на разных устройствах.

**Важно**: Все тесты должны включать проверку стилей (CSS) элементов страницы.

### Обязательные UI/UX проверки:

1. **Адаптивность на всех устройствах** (обязательно в каждом тесте):
   - Desktop (1920x1080)
   - Tablet (768x1024)
   - Mobile (375x667)

2. **Доступность (A11y)**:
   - Alt-тексты для изображений
   - Структура заголовков (h1-h6)
   - ARIA-атрибуты для интерактивных элементов
   - Доступные имена для кнопок и ссылок

3. **Визуальная проверка**:
   - Видимость основного контента
   - Кликабельность элементов
   - Корректное отображение текста
   - Отсутствие обрезанного контента

4. **Проверка стилей (CSS)** - **ОБЯЗАТЕЛЬНО В КАЖДОМ ТЕСТЕ**:
   - ✅ Проверка загрузки CSS файлов
   - ✅ Проверка применения стилей к элементам (font-size, color, padding, margin, border-radius и т.д.)
   - ✅ Проверка корректного отображения цветов, шрифтов, размеров
   - ✅ Проверка отсутствия сломанных стилей (например, элементы с display: none, когда должны быть видимы)
   - ✅ Проверка видимости элементов (display, visibility, opacity)
   - ✅ Проверка responsive стилей на разных размерах экрана
   
   **Используйте функцию `checkStyles()` из `./helpers/ui-ux-helpers` или проверяйте стили вручную для конкретных элементов.**

5. **Производительность**:
   - Время загрузки страницы
   - TTFB (Time To First Byte)
   - Время до интерактивности DOM

### Использование helper-функций:

Для автоматической проверки адаптивности используйте функцию `checkUIUX()`:

```typescript
import { checkUIUX } from './helpers/ui-ux-helpers';

await test.step('Проверка UI/UX и адаптивности', async () => {
  await checkUIUX(page);
});
```

### Стандартные viewport размеры:

```typescript
import { VIEWPORTS } from './helpers/ui-ux-helpers';

// VIEWPORTS.DESKTOP = { width: 1920, height: 1080 }
// VIEWPORTS.TABLET = { width: 768, height: 1024 }
// VIEWPORTS.MOBILE = { width: 375, height: 667 }
```

### Доступные helper-функции:

- `checkUIUX(page)` - комплексная проверка (адаптивность + доступность + стили + производительность)
- `checkResponsiveDesign(page)` - проверка адаптивности на всех устройствах
- `checkAccessibility(page)` - проверка доступности (A11y)
- `checkStyles(page)` - проверка стилей (CSS)
- `checkPerformance(page)` - проверка производительности

### Пример проверки стилей:

```typescript
import { checkStyles } from './helpers/ui-ux-helpers';

await test.step('Проверка стилей', async () => {
  await checkStyles(page);
});
```

### Что проверяет `checkStyles()`:

1. **Загрузка CSS файлов** - проверяет наличие `<link rel="stylesheet">` на странице
2. **Применение стилей к body** - проверяет наличие основных стилей (fontFamily, fontSize, color)
3. **Стили видимых элементов** - проверяет, что видимые кнопки не скрыты через CSS (display, visibility, opacity)
4. **Стили основного контента** - проверяет, что основной контент не скрыт через display: none
5. **Отсутствие сломанных стилей** - проверяет, что видимые элементы не имеют display: none

### Рекомендации по проверке стилей:

1. **Проверка загрузки CSS:**
   - Убедитесь, что CSS файлы загружены
   - Проверьте отсутствие ошибок загрузки стилей

2. **Проверка применения стилей:**
   - Используйте `window.getComputedStyle()` для проверки примененных стилей
   - Проверяйте важные свойства: `display`, `visibility`, `opacity`, `color`, `font-size`

3. **Проверка видимости элементов:**
   - Элементы, которые должны быть видимы, не должны иметь `display: none` или `visibility: hidden`
   - Проверяйте `opacity` для элементов, которые могут быть скрыты через прозрачность

4. **Проверка responsive стилей:**
   - Проверяйте стили на разных размерах экрана
   - Убедитесь, что элементы корректно адаптируются

## 🧪 Тестовые сценарии

### 1. Login Form Validation (`A Login Form Validation.spec.ts`)

**LOGIN FORM SUITE — КОМПЛЕКСНЫЙ ТЕСТ ФОРМЫ АВТОРИЗАЦИИ**

Данный набор тестов выполняет исчерпывающую проверку формы входа, включая визуальное соответствие, функциональную валидацию, безопасность (XSS/SQLi) и удобство использования (UI/UX).

**ЧЕК-ЛИСТ ПРОВЕРОК:**

1. **АВТОРИЗАЦИЯ И ОСНОВНОЙ ФУНКЦИОНАЛ:**
   - Успешный вход с валидными данными и проверка исчезновения формы
   - Автоматическая обработка лимитов на попытки входа (Rate Limiting)
   - Проверка сохранения сессии и корректности типов полей (email/password)

2. **UI/UX И ВИЗУАЛЬНЫЙ АУДИТ:**
   - Адаптивность: проверка корректного отображения на различных разрешениях
   - Доступность (Accessibility): соответствие стандартам WCAG
   - Стили: проверка CSS-параметров элементов формы
   - Видимость: контроль присутствия всех ключевых полей и кнопок

3. **ГЛУБОКАЯ ВАЛИДАЦИЯ EMAIL:**
   - Обязательность: проверка поведения при пустом поле
   - Формат (Data-Driven): тестирование различных невалидных паттернов (без @, без домена и т.д.)
   - Пробелы: проверка автоматической обрезки (trim) краевых пробелов и запрет пробелов внутри
   - Регистр: подтверждение нечувствительности email к заглавным буквам
   - Границы: проверка лимита длины согласно стандартам RFC (до 254 символов)
   - Спецсимволы: проверка разрешенных (._-+) и запрещенных символов

4. **ГЛУБОКАЯ ВАЛИДАЦИЯ ПАРОЛЯ:**
   - Обязательность: реакция системы на пустой пароль
   - Длина: проверка минимальных (6-8) и максимальных (64-128) ограничений
   - Сложность: тестирование устойчивости к паролям разного состава (регистр, цифры, символы)
   - Интерактивность: проверка функционала «Глазика» (переключение видимости пароля)
   - Ошибки: проверка корректности сообщений при неверном пароле (без раскрытия существования email)

5. **БЕЗОПАСНОСТЬ И ЗАЩИТА:**
   - Защита от XSS: попытки внедрения <script> в поля ввода
   - Защита от SQL-инъекций: тестирование классических паттернов (' OR 1=1 --)
   - Конфиденциальность: проверка отсутствия логирования пароля в открытом виде в HTML-коде

6. **ДОПОЛНИТЕЛЬНЫЕ ЭЛЕМЕНТЫ:**
   - Forgot Password: проверка наличия ссылки и корректности перехода на форму восстановления
   - Remember Me: проверка наличия чекбокса, его состояний и влияния на форму

**ИТОГ:** Тест гарантирует, что форма авторизации надежна, защищена от атак, удобна для пользователя и соответствует бизнес-требованиям.

### 2. Contractor Registration (`B Sign Up Form Contractor Registration .spec.ts`)

**CONTRACTOR REGISTRATION — КОМПЛЕКСНЫЙ АУДИТ ФОРМЫ РЕГИСТРАЦИИ ПОДРЯДЧИКА**

Данный тестовый набор выполняет исчерпывающую проверку страницы регистрации подрядчика (Contractor Registration), охватывая визуальное соответствие (UI/UX), функциональную валидацию каждого поля, безопасность и адаптивность.

**ЧЕК-ЛИСТ ПРОВЕРОК:**

1. **НАВИГАЦИЯ И ИНИЦИАЛИЗАЦИЯ:**
   - Проверка перехода с главной страницы через "Sign Up" к форме регистрации подрядчика
   - Валидация основного заголовка и корректности загрузки страницы

2. **ПОЛЕ "BUSINESS NAME" (НАЗВАНИЕ КОМПАНИИ):**
   - UI/UX: Проверка Label, обязательности (*), Placeholder, шрифтов и состояний фокуса (CSS border/outline/shadow)
   - Состояние: Проверка доступности для редактирования (not readonly/disabled) и навигации через Tab
   - Ввод данных: Поддержка Латиницы, Кириллицы, цифр, спецсимволов (&, #, №, @, кавычки, дефисы) и пробелов
   - Границы: Тесты на 1, 2, 200 (max) и 201 символ (обрезание по maxlength)
   - Валидация: Ошибка при пустом поле, очистка через Ctrl+A + Backspace, обработка только пробелов (trim)
   - Безопасность: Защита от XSS (скрипты) и SQL-инъекций

3. **ПОЛЕ "FULL NAME" (ИМЯ И ФАМИЛИЯ):**
   - Валидация форматов: Сложные имена с дефисами (Jean-Pierre), апострофами (O'Connor), титулами (Dr.) и диакритикой (René)
   - Негативные тесты: Цифры в имени, запрещенные спецсимволы (@, #, $), минимальная длина (1 vs 2 символа)
   - Технические проверки: Максимальная длина (50/100/255 символов), автоматический Trim и нормализация двойных пробелов
   - Безопасность: Проверка экранирования HTML-тегов и запрет выполнения JS-скриптов

4. **ПОЛЕ "EMAIL" (ЭЛЕКТРОННАЯ ПОЧТА):**
   - Атрибуты: Проверка type="email" (для вызова мобильной клавиатуры) и корректности autocomplete
   - Форматы: Поддержка поддоменов, символов "+" (тегирование), длинных доменов
   - Негативные тесты: Отсутствие @, отсутствие домена, некорректные символы, пробелы внутри адреса, кириллица
   - Функциональность: Обязательность заполнения, обрезка пробелов и проверка нечувствительности к регистру

5. **ПОЛЯ "PASSWORD" И "CONFIRMATION" (ПАРОЛЬ И ПОДТВЕРЖДЕНИЕ):**
   - Безопасность: Проверка маскировки символов (type="password") и наличия подсказок по длине
   - Интерактивность: Функционал иконок «Глаз» (переключение видимости) отдельно для каждого поля (изоляция)
   - Валидация: Минимум 8 символов, максимальная длина (128/255), поддержка спецсимволов и цифр
   - Логика совпадения: Проверка ошибки "Passwords do not match" при расхождении данных

6. **КНОПКА "SIGNUP" (РЕГИСТРАЦИЯ):**
   - Визуальный аудит: Стили, текст, типы, поведение при наведении (Hover) и фокусе
   - Бизнес-логика: Состояние Disabled при наличии ошибок в полях и Enabled при корректном заполнении всей формы
   - Функциональный тест: Имитация клика и проверка навигации на страницу успеха или отображение сообщения о регистрации

7. **ГЛОБАЛЬНЫЙ UI/UX И АДАПТИВНОСТЬ:**
   - Вызов комплексного хелпера checkUIUX для проверки верстки на различных разрешениях (Desktop/Tablet/Mobile)
   - Проверка семантики, доступности (Accessibility) и отсутствия визуальных артефактов

**ИТОГ:** Тест гарантирует, что форма регистрации подрядчика надежно защищена от некорректного ввода, обеспечивает высокую конверсию за счет качественного UX и технически стабильна.

### 3. Homeowner Signup Form (`C Sign Up Form Homeowner.spec.ts`)

**HOMEOWNER SIGNUP — КОМПЛЕКСНЫЙ ТЕСТ ФОРМЫ РЕГИСТРАЦИИ ДОМОВЛАДЕЛЬЦА**

Данный тестовый набор выполняет полный аудит процесса регистрации пользователя типа "Homeowner", включая навигацию, визуальное соответствие (UI/UX), функциональную валидацию полей и безопасность.

**ЧЕК-ЛИСТ ПРОВЕРОК:**

1. **НАВИГАЦИЯ И ДОСТУПНОСТЬ:**
   - Проверка наличия и кликабельности ссылки "Sign Up" в главном меню
   - Переход в раздел выбора типа регистрации и выбор опции "Homeowner Sign Up"
   - Валидация корректности заголовка формы ("Homeowner Signup")

2. **ПОЛЕ "FULL NAME" (ИМЯ И ФАМИЛИЯ):**
   - UI-атрибуты: проверка плейсхолдера, обязательности (required), типа (text) и автозаполнения (name)
   - Стили: аудит шрифтов, отступов и состояний фокуса (изменение цвета границ и теней)
   - Взаимодействие: работа клавиши Tab, очистка поля через горячие клавиши (Ctrl+A + Backspace)

### Важно: Проверка стилей

**ОБЯЗАТЕЛЬНО**: В каждом новом тесте должна быть проверка стилей элементов страницы.

Проверка стилей включает:
- ✅ Проверку загрузки CSS файлов
- ✅ Проверку применения стилей к элементам (font-size, color, padding, margin и т.д.)
- ✅ Проверку видимости элементов (display, visibility, opacity)
- ✅ Проверку отсутствия сломанных стилей
- ✅ Проверку responsive стилей на разных размерах экрана

**Используйте функцию `checkStyles()` из helper-файла:**

```typescript
import { checkStyles } from './helpers/ui-ux-helpers';

await test.step('Проверка стилей', async () => {
  await checkStyles(page);
});
```

**Или проверяйте стили вручную для конкретных элементов:**

```typescript
await test.step('Проверка стилей заголовка', async () => {
  const heading = page.getByRole('heading', { name: /Account Settings/i });
  await expect(heading).toBeVisible();
  
  const styles = await heading.evaluate((el) => {
    const computed = window.getComputedStyle(el);
    return {
      fontSize: computed.fontSize,
      fontWeight: computed.fontWeight,
      color: computed.color,
    };
  });
  
  expect(styles.fontSize).toBe('24px');
  expect(styles.fontWeight).toBe('500');
  console.log(`✅ Стили заголовка: font-size=${styles.fontSize}, font-weight=${styles.fontWeight}`);
});
```

### Важно: Закрытие браузера после теста

**Каждый тест должен явно закрывать браузер после завершения!**

Используйте `test.afterEach` для гарантированного закрытия браузера и очистки ресурсов:

```typescript
test.afterEach(async ({ page, context }) => {
  // Закрываем все открытые страницы
  await page.close();
  // Закрываем контекст браузера
  await context.close();
});
```

**Альтернативный вариант** (если используете `browser` fixture):

```typescript
test.afterEach(async ({ browser }) => {
  // Закрываем все контексты браузера
  if (browser) {
    await browser.close();
  }
});
```

**Почему это важно:**
- ✅ Освобождает ресурсы системы
- ✅ Предотвращает утечки памяти
- ✅ Обеспечивает чистоту между тестами
- ✅ Улучшает стабильность тестов
- ✅ Гарантирует корректное завершение тестов
